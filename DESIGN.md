## 一、项目背景

### 1.1 业务背景

随着我行各项业务的持续发展，各类业务系统所依托的数据库规模已接近 **200 套**，涵盖 Oracle、MySQL、PostgreSQL、达梦（DM8）、Gbase 8a、TDSQL 等多种异构数据库类型，分布于生产、同城灾备、异地灾备等多套环境中。

数据库作为业务系统的核心数据载体，其运行稳定性直接关系到业务连续性与客户服务质量。然而，当前数据库运维工作仍依赖**人工手工运维模式**，在维护对象规模持续扩大、业务复杂度不断提高的背景下，该模式已暴露出明显的结构性瓶颈，难以支撑银行业务系统对数据库高可用性、高性能的持续要求。

### 1.2 建设目标

计划建设**数据库自动化智能运维平台（DB-AIOps）**，以"数据驱动、智能分析、主动运维"为核心理念，实现以下三个层次的目标：

**目标一：智能容量管理**
能够对数据库常规量化指标进行智能监控记录，分析量化指标的使用情况，并依据使用情况预测未来一段时间的增长周期和资源剩余量，从而智能地进行扩容建议和资源管理，取代机械式阈值告警。

**目标二：性能偏移感知与业务影响研判**
能够对数据库各项性能指标进行长期记录，并根据采集记录信息，自主研判各类性能指标发生较大范围偏移波动时，对业务产生的影响进行智能判断，并给出处置建议。

**目标三：智能决策辅助**
能够针对数据库运维过程中需要 DBA 凭经验判断的疑难情况，或需要权衡利弊的复杂操作，通过对整个数据库的长期分析，给出未来的运行资源分配和使用规划，同时根据运行情况给出性能和稳定性方面的优化建议。

## 二、现状与痛点分析

### 2.1 运维现状

| 维度 | 现状描述 |
|------|----------|
| 规模 | 数据库实例近 200 套，Oracle/MySQL/PostgreSQL/达梦/Gbase 8a/TDSQL 六种类型并存，分布于生产、灾备等多环境 |
| 人员 | 专职数据库运维人员数量有限，人均维护实例数量大，精力分散 |
| 工具 | 以数据库原生管理工具（SQL*Plus、pgAdmin、Navicat 等）为主，缺乏统一监控视图和集中管理入口 |
| 巡检 | 依赖人工脚本和个人经验，巡检频率低（通常以天为单位），无法 7×24 全量覆盖 |
| 告警 | 基于固定阈值的简单告警（若有），缺乏对业务模式的感知，误报率高、真实问题发现滞后 |
| 应急 | 故障应急响应路径长，缺少系统性的根因分析辅助，定位过程高度依赖个人经验 |
| 知识 | 运维经验沉淀在个人，缺乏系统性知识积累和传承机制 |

### 2.2 核心痛点

**痛点一：监控覆盖率不足**

人工维护力量有限，无法对 200 套数据库进行高频次、全维度的巡检覆盖，存在明显的监控盲区。关键性能指标（连接数、表空间、锁等待等）的采集频率往往以天为单位，问题发现严重滞后，错失最佳处置窗口期。

**痛点二：告警机制质量低**

若干基于固定阈值的告警存在两类问题：
- **阈值过低**：正常业务高峰（如月末批量、节假日）也触发告警，运维人员因频繁"狼来了"形成告警疲劳，真实告警被淹没
- **阈值过高**：问题已影响业务才触发告警，丧失主动干预机会

固定阈值无法感知业务时间模式（如"周一早 9 点连接数高是正常的"），更无法区分"短暂抖动"与"持续劣化"。

**痛点三：运维模式被动**

当前的故障发现路径通常为：**业务方反馈系统异常 → 运维排查 → 向下追溯到数据库层面故障**。数据库层面的问题往往在已对业务产生可感知影响后才被发现，不能在萌芽阶段主动介入。

**痛点四：经验依赖难以复制**

数据库优化调整（SQL 优化、参数调整、容量规划、锁分析等）高度依赖个人经验，知识无法系统性沉淀。当面对疑难问题或需要权衡利弊的复杂操作时，缺乏数据支撑和辅助决策工具，完全依赖资深 DBA 的判断。人员变动带来较大的运维连续性风险。

**痛点五：容量管理粗放**

容量管理依赖阈值触发（如表空间使用率 > 90%），本质是事后干预：当阈值触发时，留给运维人员的处置时间窗口已经很短，甚至可能需要紧急扩容，带来业务中断风险。无法基于数据增长趋势提前规划，也无法给出量化的扩容建议。

**痛点六：运维操作缺乏管控**

高风险运维操作（KILL SESSION、扩容、DDL 变更等）缺少统一的审批流程和执行记录，操作可追溯性差，不符合金融机构的合规管理要求。

## 三、需求分析

### 3.1 需求分析框架

三个建设方向在能力层次上呈递进关系，后一层依赖前一层积累的数据和能力：

```
┌─────────────────────────────────────────────────────────────┐
│  方向三：运维决策辅助                                         │
│  依赖：长期历史数据 + 方向一/二的分析结果                     │
│  输出：资源规划报告、SQL优化建议、健康评分、参数调整建议       │
├─────────────────────────────────────────────────────────────┤
│  方向二：性能偏移感知与业务影响研判                            │
│  依赖：连续的历史指标数据（方向一的积累）                     │
│  输出：动态基线、偏移告警、根因分析、业务影响评估             │
├─────────────────────────────────────────────────────────────┤
│  方向一：智能量化监控与容量管理                               │
│  依赖：数据库采集能力（当前已具备雏形）                       │
│  输出：指标采集、趋势分析、增长预测、扩容建议                 │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 方向一：智能量化监控与容量管理需求

#### 3.2.1 指标采集需求

**采集对象**：全行近 200 套数据库，覆盖 Oracle、MySQL、PostgreSQL、达梦 DM8、Gbase 8a、TDSQL。

**采集频率要求**：

| 场景 | 采集频率 | 说明 |
|------|----------|------|
| 正常状态 | 5 分钟/次 | 日常运行监控 |
| 指标异常（偏离基线） | 1 分钟/次 | 自动加密采集，追踪异常演变 |
| 数据库 DOWN | 30 秒/次 | 高频探活，快速感知恢复 |
| 业务高峰期（可配置） | 1 分钟/次 | 重要时间段主动提升采集精度 |

**核心采集指标分类**：

| 指标类别 | 指标项 | 适用数据库 |
|----------|--------|------------|
| 可用性 | 连通状态、连接响应时间（毫秒） | 全部 |
| 连接 | 当前连接数、最大连接数限制、连接使用率（%） | 全部 |
| 容量-表空间 | 各表空间总量（MB）、已用量（MB）、使用率（%）、本次采集增量 | Oracle/DM8 |
| 容量-数据库大小 | 各数据库总大小（MB）、本次采集增量 | MySQL/PG/Gbase |
| 吞吐 | QPS（每秒查询数）、TPS（每秒事务数） | MySQL/PG/Oracle |
| 慢查询 | 慢查询累计数量、本次采集周期内新增慢查询数 | MySQL/PG/Oracle |
| 锁等待 | 当前锁等待数量、最长等待时长（秒）、阻塞会话信息 | 全部 |
| 缓存 | Buffer Pool/Buffer Cache 命中率（%） | MySQL/Oracle |
| 日志 | Redo/Binlog 生成速率（MB/小时） | MySQL/Oracle/PG |
| 等待事件 | Top 5 等待事件名称及时间占比（%） | Oracle |
| 集群 | 节点状态（在线/离线）、角色（主/从） | Gbase/TDSQL |
| 实例 | 数据库版本、实例启动时间（uptime 秒数） | 全部 |

**采集可靠性要求**：
- 每个数据库的采集任务有独立的超时控制（默认 10 秒），超时后记录 DOWN 状态，不影响其他库的采集
- 采集 SQL 全部为只读操作，严禁在生产库执行任何写操作
- 采集账号使用最小权限原则（仅授予监控所需的系统视图查询权限，清单见附录）

#### 3.2.2 增长趋势预测需求

系统应能基于历史指标数据，自动识别以下四种增长模式并选择对应的预测模型：

| 增长模式 | 特征描述 | 适用预测模型 | 典型场景 |
|----------|----------|--------------|----------|
| 线性稳定增长 | 数据以固定速率持续增长 | 线性回归（OLS） | 业务稳定期的数据积累 |
| 带周期的增长 | 有月末/季末/年末高峰叠加在增长趋势上 | Holt-Winters 三次指数平滑 | 金融业务特征明显的库 |
| 波动增长 | 增长趋势存在但增速不稳定 | ARIMA | 业务变化频繁的库 |
| 数据不足 | 接入时间 < 30 天，历史数据不足以建模 | 简单线性外推（降级） | 新接入数据库 |

**预测结果的必要输出项**：
- 预计触达告警线（默认 85%）的日期
- 预计触达危险线（默认 95%）的日期
- 当前月均增长率（%/月）
- 预测置信度（0-1）和所用模型名称
- 推荐扩容时间窗口和扩容量建议（含安全余量）

#### 3.2.3 主动容量管理需求

- 系统应在预计触达告警线的 **30 天前**（可按库配置）自动生成容量扩容建议，作为运维工单进入审批流程
- 扩容建议工单须包含：建议扩容量（基于预测增量 × 1.5 的安全系数）、建议执行时间窗口、扩容操作 SQL 草稿
- 支持按季度、年度自动汇总生成资源需求预测报告，供管理层用于 IT 资源采购计划

### 3.3 方向二：性能偏移感知与业务影响研判需求

#### 3.3.1 性能指标长期记录需求

- 对 3.2.1 中所有性能类指标保留**至少 1 年**的原始采集数据
- 在原始数据基础上，按 1 小时和 1 天两个粒度进行预聚合存储（最大值/最小值/均值/P95），支持历史趋势的快速查询
- 历史数据须支持导出，格式为 CSV 或 Excel

#### 3.3.2 动态基线建模需求

**核心设计理念**：建立"时间感知的正常行为模型"，使告警系统理解"周一早 9 点连接数峰值"与"周三凌晨 3 点连接数峰值"属于截然不同的两种情况。

**基线维度模型**：

```
基线单元 = (数据库实例 × 指标名 × 时间槽)

时间槽划分（共 168 个槽）：
  一周 7 天 × 每天 24 小时 = 168 个独立的基线单元

扩展时间分类（可配置）：
  - 工作日基线 / 节假日基线（需维护节假日日历）
  - 月末最后3天特殊基线
  - 批量作业时段基线（DBA 可标记特定时间段为"批量时段"）
```

**基线计算规格**：
- 采用滑动窗口：使用近 28 天（默认，可配置 7/14/28/60 天）的同一时间槽历史数据
- 计算统计量：均值（μ）、标准差（σ）、P90、P95、P99
- 正常范围：`[μ - k×σ, μ + k×σ]`，k 值按指标类型配置（默认 k=2）
- 异常值剔除：计算基线前自动剔除历史数据中已被标记为异常的数据点，防止污染基线
- 数据不足时的降级策略：
  - 同一时间槽历史数据 < 7 个点时，使用全天候统计数据代替时间槽统计
  - 全天候历史数据 < 24 小时时，标记为"基线未就绪"，不进行基线告警，仅使用固定阈值兜底

#### 3.3.3 偏移量智能判定需求

**三重条件告警机制**（三个条件须同时满足方触发告警，避免误报）：

| 条件维度 | 判定规则 | 说明 |
|----------|----------|------|
| 量级 | 当前值超出正常范围 `[μ-2σ, μ+2σ]` 为警告；超出 `[μ-3σ, μ+3σ]` 为严重 | 依赖基线统计量 |
| 方向 | 上升敏感指标（连接数、慢查询、锁等待）只在异常升高时告警；下降敏感指标（缓存命中率、TPS 骤降）只在异常下降时告警 | 按指标配置，双向敏感指标两个方向均告警 |
| 持续性 | 连续 **3 次**采集均满足量级和方向条件（约 15 分钟）才触发，单次异常不告警 | 过滤瞬时抖动 |

**告警恢复机制**：
- 当偏离指标连续 2 次采集恢复至正常范围内，自动触发"告警恢复"通知
- 告警期间不重复发送相同告警（告警静默），只在新异常首次出现或状态变化时发送

#### 3.3.4 业务影响评估需求

**数据维护**（由 DBA/管理员配置）：
- 维护数据库实例与上层业务系统的多对多关联关系
- 每套业务系统标记重要程度：`核心（Core）/ 重要（Important）/ 一般（Normal）`
- 核心业务系统可配置依赖链（如：核心交易系统依赖主库 + 只读从库 + 缓存 + 消息队列）

**评估逻辑**：当某数据库触发 P1/P2 级告警时，系统自动生成业务影响评估报告，内容包括：
- 受影响的关联业务系统列表（按重要程度排序）
- 影响程度评级（`严重/中等/轻微`，取关联业务系统的最高重要程度映射）
- 影响机理描述（结合告警类型的文字说明，如"连接数耗尽可能导致新事务被拒绝"）
- 推荐处置措施（来自 RCA 规则库的建议）

### 3.4 方向三：智能决策辅助需求

#### 3.4.1 数据库综合健康评分需求

系统应按日对每个数据库生成综合健康评分（满分 100 分），评分须有量化依据，各维度分项可钻取：

| 评分维度 | 权重 | 评分依据 |
|----------|------|----------|
| 可用性（Availability） | 25% | 近 30 天内停机次数、累计停机时长、最近一次恢复时长 |
| 容量（Capacity） | 25% | 最紧张资源的剩余量及预测到期时间 |
| 性能（Performance） | 25% | 近 7 天性能指标相对基线的偏离程度和频次 |
| 配置（Configuration） | 15% | 关键参数合理性检查得分 |
| 运维规范（Operations） | 10% | 未处理的活跃告警数、待审批工单数、过期统计信息数 |

评分变化趋势（日环比、周环比）同步展示，支持识别评分下降趋势。

#### 3.4.2 资源使用规划需求

- 基于 6-12 个月的历史数据，为每个数据库生成**资源使用画像**，识别数据库负载特征（高读/高写/高并发/OLAP 分析型等）
- 识别资源使用效率极端情况：
  - **严重过剩**：长期资源使用率 < 20%，建议整合或缩减
  - **资源紧张**：长期资源使用率 > 70%，建议规划扩容
- 支持生成**年度资源规划报告**，汇总各库的存储、连接等资源预测，辅助 IT 基础设施采购决策

#### 3.4.3 性能优化建议需求

| 建议类型 | 触发条件 | 建议内容示例 |
|----------|----------|--------------|
| 索引建议 | 采集到高频全表扫描 SQL（慢查询日志 / AWR 数据） | "发现 SQL [hash] 每日执行 50 万次，全表扫描 orders 表，建议为 (user_id, status) 建立组合索引" |
| 参数调整建议 | Buffer Pool 命中率持续低于阈值，或内存参数与工作负载明显不匹配 | "InnoDB Buffer Pool 命中率近 7 天均值 87%，低于建议值 95%，当前 4GB，建议调整为 8GB" |
| 连接池建议 | 连接数使用率均值持续 > 70% | "近 30 天连接数均值 240/300，建议评估引入 PgBouncer/ProxySQL 连接池中间件" |
| 维护提醒 | 统计信息更新时间超过 N 天，或索引碎片率超过阈值 | "表 trade_order 的统计信息已 45 天未更新（上次: 2026-03-01），建议执行 GATHER STATS" |
| 架构优化建议 | 识别到读写分离未利用、大表归档机会等结构性问题 | "历史订单表 order_2024 已占用 120GB，建议评估归档策略" |

所有建议须附有量化数据依据，支持"接受（生成工单）"或"忽略"操作，忽略需填写原因。

### 3.5 非功能性需求

| 类别 | 需求项 | 指标要求 |
|------|--------|----------|
| **性能** | 采集并发能力 | 支持 200 套数据库并发采集，单轮全量采集 < 5 分钟完成 |

### 5.1 架构设计原则

| 原则 | 说明 |
|------|------|
| 分层解耦 | 采集、存储、分析、展示各层职责单一、接口清晰，可独立扩展和替换 |

#### 6.1.2 采集结果标准 Schema

所有 Checker 的 `collect_metrics` 返回值须包含以下核心字段（各库特有指标作为扩展字段附加）：

      "blocker_id":   "string（阻塞方会话标识）",

      "waiter_id":    "string（被阻塞方会话标识）",

    }

      "total_mb":  "float（总容量 MB）",
      "used_mb":   "float（已用 MB）",

```

> **注**：`used_pct` 必须表示该存储对象相对其物理容量上限的使用比例，不得使用相对最大值的自比较算法（当前 PostgreSQL Checker 中存在此 Bug，详见第一期修复清单）。

#### 6.1.3 自适应采集调度设计

```
采集调度器维护每个数据库的"当前采集状态"：
  - NORMAL:    5 分钟间隔
  - ELEVATED:  1 分钟间隔（触发条件：任一指标偏离基线 2σ 以上）
  - RECOVERY:  30 秒间隔（触发条件：数据库由 UP 变为 DOWN 后，直至恢复）
  - SCHEDULED_HIGH: 1 分钟间隔（DBA 手动设置业务高峰监控时段）

状态转换：
  ELEVATED 连续 10 次采集均无异常 → 自动回退到 NORMAL
  RECOVERY 检测到数据库 UP → 发送恢复通知 → 回退到 NORMAL
```

### 6.2 动态基线引擎

#### 6.2.1 基线建模流程

```
每日凌晨 2:00（低峰期）执行：

FOR EACH 活跃数据库 × 可建模指标:
  1. 读取近 28 天（可配置）历史数据，按时间槽分组（168 槽）
  2. 剔除被标记为异常的数据点
  3. 统计每个时间槽：count / mean / std / p90 / p95 / p99
  4. 若某槽数据量 < 7，标记为"数据不足"，使用全时段统计值作为该槽的基线
  5. 计算 normal_min = mean - k*std, normal_max = mean + k*std（k 按指标配置）
  6. 将计算结果写入 BaselineModel 表（upsert，更新 updated_at）
  7. 刷新 Redis 中该数据库×指标的基线缓存
```

#### 6.2.2 实时异常检测流程

```
每次采集完成后，对每个 UP 状态的指标执行：

1. 从 Redis 读取该 (db_id × metric × 当前时间槽) 的基线
2. 若基线未就绪 → 退出，使用固定阈值兜底检查
3. 检查量级：当前值是否超出 [normal_min, normal_max]
4. 检查方向：超出方向是否与该指标的"告警方向"配置一致
5. 检查持续性：查询近 3 次采集记录，是否均满足 3/4 步条件
6. 若三重条件全满足：
   a. 查询上一次告警记录（同库×同指标×同方向）
   b. 若上次告警已恢复（status=resolved）或不存在 → 创建新 Alert，触发推送
   c. 若上次告警仍活跃（status=active）→ 不重复推送（防告警风暴）
7. 若当前值已恢复正常，但存在活跃告警 → 更新 Alert status=resolved，触发恢复通知
```

### 6.3 容量预测引擎

#### 6.3.1 模型选择逻辑

```
Input: 某表空间/数据库 近 N 天的每日末 used_pct 序列

Step 1: 数据充分性检查
  - 历史数据 < 14 天 → 降级为简单线性外推
  - 历史数据 14-30 天 → 只使用线性回归
  - 历史数据 > 30 天 → 继续 Step 2

Step 2: 周期性检测（使用 FFT 或自相关分析）
  - 存在显著 7 天周期 → 候选模型加入 Holt-Winters
  - 存在显著 30 天周期 → 候选模型加入 Holt-Winters（年周期型）

Step 3: 对候选模型用最近 7 天数据做交叉验证，选 MAPE 最小的模型

Step 4: 使用选定模型预测未来 90 天的 used_pct 序列

Step 5: 从预测序列中提取：
  - 预计超过 warn_threshold（默认85%）的日期
  - 预计超过 critical_threshold（默认95%）的日期
  - 月均增长率
  - 95% 置信区间上下界

Step 6: 结果写入 PredictionResult 表，触发规则检查
  - 若 warn_date 距今 < 30 天且尚未生成有效工单 → 自动创建扩容建议工单
```

#### 6.3.2 扩容量计算规则

```
推荐扩容量 = 预计未来 90 天增量 × 安全系数（默认 1.5）
扩容到目标 = (current_used + 推荐扩容量) / target_threshold（默认0.70）

扩容后目标使用率建议 ≤ 70%（为未来 3 个月增长留有余量）
```

### 6.4 根因分析引擎（RCA）

规则驱动的事件-原因-建议三元组知识库，支持规则热加载（无需重启服务）：

| 规则 ID | 触发现象 | 推断原因 | 处置建议 |
|---------|----------|----------|----------|
| R001 | 连接数连续增长，超基线 3σ 以上 | 连接泄漏（应用未正确释放连接） | 检查应用连接池配置；执行 SQL 定位未关闭连接的来源 IP 和应用程序 |
| R002 | 慢查询数量突增（超基线 3σ）且 QPS 同步下降 | 执行计划突变（统计信息过期 / 绑定变量未使用） | 立即收集统计信息；检查近期是否有大批量数据变更导致统计信息失效 |
| R003 | 锁等待数量上升，涉及特定表的 DML 操作 | 热点表行锁竞争（大事务未提交 / 批量操作未拆分） | 定位阻塞源 SQL；检查是否有大事务；考虑拆分批量操作为小批次 |
| R004 | 表空间使用率在短时间内（< 1 小时）快速上升 ≥ 5% | 数据膨胀（误操作 / 业务 Bug / 日志或临时表爆增） | 立即查找大对象增长情况；检查最近 1 小时内的大 DML 操作 |
| R005 | QPS/TPS 骤降（超基线 3σ），无对应的锁等待 | I/O 子系统性能瓶颈 | 检查存储层 I/O 延迟和吞吐；排查是否有全表扫描大查询 |
| R006 | Buffer Pool/Buffer Cache 命中率持续低于 90% | 内存不足或存在大表全扫描污染 | 检查是否有全表扫描 SQL 将热数据从缓冲池中驱逐；评估增大缓冲池参数 |
| R007 | Redo/Binlog 生成速率突增（超基线 5 倍以上） | 计划外的大批量 DML 操作 | 确认是否有计划内批量作业；若非计划内，立即定位来源会话 |
| R008 | 数据库由 UP 变为 DOWN | 宕机 / 手动重启 / OOM Kill | 立即检查操作系统日志和数据库 Alert Log；确认是否为计划内操作 |
| R009 | 连接数使用率 > 80%，且连接来自少数几个来源 IP | 特定应用连接池配置过大或连接泄漏 | 按来源 IP 分析连接分布；限制问题 IP 的最大连接数 |
| R010 | 等待事件 Top 1 为 "log file sync"（Oracle） | Redo Log 写入成为瓶颈 | 检查 Redo Log 所在磁盘 I/O；考虑将 Redo Log 迁移到更快的存储 |

**复合因果链推导**：当多条规则同时触发时，系统尝试推断关联关系，例如：
- R001（连接泄漏）+ R003（锁等待）→ 推断为"连接堆积引发锁竞争"复合故障，处置优先级提升为 P1

### 6.5 告警管理模块

#### 6.5.1 告警级别与处置要求

| 级别 | 说明 | 推送方式 | 响应时限 |
|------|------|----------|----------|
| P1 严重 | 数据库实例 DOWN / 连接数耗尽（使用率 > 95%）/ 容量触达危险线 | 短信 + 钉钉/企微 + 邮件 | 15 分钟内响应 |
| P2 重要 | 性能指标偏离基线 3σ / 容量触达告警线 / 锁等待持续 > 5 分钟 / 预测 7 天内触达危险线 | 钉钉/企微 + 邮件 | 1 小时内响应 |
| P3 警告 | 性能指标偏离基线 2σ / 容量使用率 > 80% / 预测 30 天内触达告警线 | 邮件 | 当日内响应 |
| P4 提示 | 优化建议 / 预测性提醒（30天以上）/ 维护提醒 | 日报汇总（邮件） | 下个维护窗口处理 |

#### 6.5.2 告警风暴防护机制

- **同实例同类告警合并**：同一数据库实例、同一告警类型，活跃期间不重复推送，仅更新告警状态
- **批量告警聚合**：同一时间窗口（5 分钟内）若有 3 个以上数据库触发同类告警（如同时多库 DOWN），合并为一条批量告警推送
- **静默窗口**：DBA 可为指定数据库设置维护期静默（如每月 22:00-02:00 的维护窗口），静默期内告警仅记录不推送
- **告警确认机制**：DBA 可对活跃告警执行"已知晓"操作，标记后该告警在 N 小时内不再提示（但不代表关闭）

### 6.6 运维操作审批模块

#### 6.6.1 风险分级与审批要求

| 风险等级 | 操作类型示例 | 审批要求 | 执行方式 |
|----------|-------------|----------|----------|
| 低风险 | 统计信息更新、会话查询、告警确认 | 无需审批，DBA 直接执行 | 平台自动执行或 DBA 手动 |
| 中风险 | Kill Session、参数临时修改（不重启）、表空间扩容 | DBA 主管审批（1 人） | 审批通过后平台自动执行 |
| 高风险 | 数据文件操作、索引 DROP、关键参数修改（需重启） | DBA 主管 + 业务负责人审批（2 人） | 审批通过后由 DBA 手动执行，结果回填 |
| 极高风险 | 生产数据 DELETE/TRUNCATE/DROP TABLE | 双 DBA 审批 + 部门负责人审批（3 人）+ 备份确认截图 | 仅平台生成 SQL 草稿，执行须在双人旁站下手动操作 |

#### 6.6.2 工单标准内容

每条运维工单须包含：
- 触发来源（手动创建 / 智能建议 / 告警自动生成）
- 操作描述（业务语言，说明"做什么、为什么"）
- 执行 SQL 完整内容（允许审批人修改）
- 风险说明（自动生成）
- 回滚方案（SQL 或操作步骤）
- 执行前检查项（清单，DBA 逐项确认）
- 预计影响范围和影响时长

## 九、安全设计

### 9.1 认证与授权

**用户角色（RBAC）**：

| 角色 | 权限范围 | 典型用户 |
|------|----------|----------|
| 只读观察员 | 查看所有数据库监控数据、告警、报告；不可执行任何操作 | 业务负责人、审计人员 |
| DBA 操作员 | 只读观察员权限 + 创建工单 + 执行低风险操作 + 确认告警 | 普通 DBA |
| DBA 主管 | DBA 操作员权限 + 审批中/高风险工单 + 管理数据库配置 | DBA 组长 |
| 系统管理员 | 全部权限 + 用户管理 + 系统配置（不包含具体数据库操作） | 平台管理员 |

**数据权限隔离**：支持按数据库分组（如按业务域或按环境）的数据范围控制，DBA 只能看到被授权的数据库组。

### 9.2 密码安全

- 所有数据库连接密码采用 **AES-256-GCM** 对称加密存储，密钥由系统管理员通过环境变量或密钥管理服务注入，不存储在代码库中
- 禁止在日志、API 响应、页面中输出任何明文密码
- 支持密码定期轮换：密码更新后平台自动使用新密码重新建立采集连接

### 9.3 审计要求

- 所有对数据库配置的修改（增删改密码）记录审计日志
- 所有运维操作工单（含拒绝、修改、执行）完整记录操作人、时间、操作内容、执行结果
- 告警的确认、静默操作记录操作人和原因
- 审计日志不可删除、不可修改，只允许追加写入
- 审计日志中的敏感信息（如密码、凭证）自动脱敏处理

## 十三、风险分析与应对

| 风险编号 | 风险描述 | 可能性 | 影响程度 | 应对措施 |
|----------|----------|--------|----------|----------|
| R01 | 历史数据积累不足，基线模型冷启动期误报率高 | 高（第一期初期必然） | 中 | 第一期积累 ≥ 4 周数据后再开放基线告警功能；冷启动期使用固定阈值兜底，明确标注"基线建立中" |
| R02 | 部分数据库采集所需系统视图权限无法获取（如 DBA_DATA_FILES 等） | 中 | 中 | 第一期前完成各类数据库最小权限清单梳理，提前与业务方协商授权；无法授权的视图采集降级处理，不影响连通性监控 |
| R03 | 采集 SQL 对生产数据库产生性能影响 | 中 | 高 | 所有采集 SQL 在测试环境进行性能评估（执行计划审核、执行耗时测试）；采集时间戳对齐，避免在同一秒内并发大量查询 |
| R04 | 告警通知渠道（邮件/钉钉）不稳定导致漏报 | 低 | 高 | 至少配置两路独立告警渠道；告警发送失败后有重试机制；关键 P1 告警在告警中心界面始终显示，不依赖推送 |
| R05 | 200 套数据库采集并发量导致平台性能问题 | 中 | 中 | 采集调度器使用线程池并发（默认 20 线程）；单轮采集超时后告警，不阻塞下一轮；若性能不足，水平扩展采集服务器 |
| R06 | 数据库密码等敏感信息安全泄露 | 低 | 极高 | 密码 AES-256 加密；密钥通过环境变量注入，不入库；平台服务器严格访问控制；定期安全审计 |
| R07 | 预测模型准确性不足，扩容建议时机偏差大 | 中 | 低 | 预测结果始终附带置信度说明；DBA 审批扩容工单时可调整建议量；持续跟踪预测偏差，定期优化模型 |
| R08 | 多类型数据库版本差异导致采集 SQL 兼容性问题 | 高 | 低 | 各 Checker 按主要版本区间做兼容性适配；采集失败时捕获异常、记录错误、标记 DOWN，不抛出到上层调度 |
| R09 | 行内网络防火墙策略限制采集服务器连接数据库 | 中 | 高 | 在平台建设立项时同步提交网络策略申请；提前与网络团队确认采集服务器到各库服务器的端口连通性 |
| R10 | 平台数据库（PostgreSQL/TimescaleDB）自身故障导致平台不可用 | 低 | 高 | 配置 PostgreSQL 主从流复制，主库故障时手动切换从库；关键配置数据定期备份 |

GRANT SELECT ON DBA_TAB_STATISTICS TO db_monitor; -- 用于统计信息检查（第三期）
```

**MySQL**（监控账号 `db_monitor@'采集服务器IP'`）：
```sql
GRANT PROCESS ON *.* TO 'db_monitor'@'采集服务器IP';
GRANT REPLICATION CLIENT ON *.* TO 'db_monitor'@'采集服务器IP';
GRANT SELECT ON performance_schema.* TO 'db_monitor'@'采集服务器IP';
GRANT SELECT ON information_schema.* TO 'db_monitor'@'采集服务器IP';
-- 第三期补充：
GRANT SELECT ON mysql.slow_log TO 'db_monitor'@'采集服务器IP';
```

**PostgreSQL**（监控账号 `db_monitor`）：
```sql
CREATE USER db_monitor WITH PASSWORD 'encrypted_password';

GRANT CONNECT ON DATABASE postgres TO db_monitor;
```

### 附录 C：名词说明
