# 数据库自动化智能运维平台
# 总体与详细设计文档（整合版）

## 1. 文档说明

### 1.1 文档目标

本文件用于统一描述数据库自动化智能运维平台的建设方案，覆盖：

- 高阶设计：建设目标、总体架构、能力边界、演进路线
- 详细设计：模块设计、数据设计、流程设计、接口设计、安全与运维设计

该版本基于当前项目现状（Django + 多数据库采集 + 告警 + 基线雏形）与业务建设诉求进行整合，目标是“能上线、可持续、可扩展”。

### 1.2 适用范围

- 适用组织：银行数据库运维团队、DBA 团队、平台运维团队
- 适用对象：约 200 套异构数据库（Oracle/MySQL/PostgreSQL/DM8/Gbase/TDSQL）
- 目标周期：12-18 个月建设窗口

---

## 2. 业务背景与核心诉求

随着业务持续增长，数据库规模接近 200 套，传统人工巡检模式出现明显瓶颈：

- 覆盖不足：人工巡检频率低、盲区多
- 响应滞后：多数问题在业务受影响后才反向定位
- 经验依赖：复杂问题高度依赖资深 DBA，稳定性受人员能力影响
- 管理粗放：以固定阈值告警为主，误报高、预判弱

### 2.1 建设目标（对应三大方向）

1. **智能量化监控与容量管理**
   - 自动采集核心指标并长期沉淀
   - 基于趋势预测容量风险，提前给出扩容建议

2. **性能偏移感知与业务影响研判**
   - 建立动态基线，识别异常偏移
   - 输出业务影响范围和处置建议，降低误报与漏报

3. **智能决策辅助**
   - 沉淀专家经验，形成 RCA 与优化建议能力
   - 支撑资源规划、稳定性治理与运维决策

## 3. 高阶设计

### 3.1 建设原则

- **稳定优先**：先保证采集和告警稳定，再迭代智能能力
- **可解释优先**：规则引擎先行，模型能力渐进引入
- **闭环优先**：监控 -> 分析 -> 建议 -> 审批 -> 执行 -> 验证
- **安全合规优先**：密钥、审批、审计全链路可追溯
- **演进优先**：单体可用、架构可拆、能力可横向扩展

### 3.2 能力边界

平台负责：

- 数据库配置管理、指标采集、告警与通知
- 基线分析、趋势预测、RCA 建议、工单审批流程
- 可视化看板、报表输出、审计追踪

平台不直接负责：

- 数据库安装升级
- 业务 SQL 代码改造（仅给建议）
- 备份恢复系统本体（可对接状态）

### 3.3 总体架构（高阶）

采用“三平面 + 分层模块”架构：

1. **控制平面**
   - 配置管理、权限策略、审批流程、系统参数

2. **数据平面**
   - 指标采集、时序写入、实时检测、告警路由

3. **智能平面**
   - 基线建模、异常评分、容量预测、RCA、健康评分

运行组件建议：

- `web-api`：Django Web/API
- `collector-worker`：采集任务执行
- `analyzer-worker`：基线/预测/评分任务执行
- `notify-worker`：通知分发与重试
- `scheduler`：任务编排（推荐 Celery Beat）

### 3.4 高阶数据流

1. 加载活跃数据库配置
2. 并发采集指标并标准化
3. 写入采集快照和时序指标
4. 执行异常检测（规则 + 基线）
5. 生成或更新告警事件
6. 路由通知（邮件/钉钉等）
7. 触发 RCA 与处置建议
8. 生成工单，进入审批执行流程
9. 执行后验证并回写审计结果

### 3.5 目标指标（SLO）

- 采集成功率（日）：>= 99.5%（剔除目标库不可用）
- 5 分钟全量采集完成率：>= 95%
- P1 告警送达延迟（P95）：<= 60 秒
- Web/API 可用性：>= 99.9%
- 基线告警误报率：<= 12%
- 容量预测 30 天 MAPE：<= 15%

## 4. 详细设计：功能模块

### 4.1 数据采集模块

#### 4.1.1 插件化采集器

统一 `BaseDBChecker` 接口，各数据库实现独立 Checker。

标准输出字段（核心）：

- `active_connections`
- `max_connections`
- `conn_usage_pct`
- `uptime_seconds`
- `qps`
- `locks[]`
- `tablespaces[]` 或 `database_sizes[]`

#### 4.1.2 并发与超时控制

- 单库超时：10-15 秒（可配置）
- 并发线程：20-40（按环境压测调优）
- 单库失败不影响全局，失败快照必须落库

#### 4.1.3 自适应采集频率

- NORMAL：5 分钟
- ELEVATED：1 分钟（出现异常偏移）
- RECOVERY：30 秒（DOWN 后探活）
- 维护高峰时段可手工提升频率

### 4.2 告警管理模块

#### 4.2.1 告警分级

- P1：DOWN、连接耗尽、严重锁等待
- P2：严重偏移、容量紧急
- P3：一般偏移、趋势风险
- P4：建议提醒类

#### 4.2.2 告警状态机

- `active` -> `acknowledged` -> `resolved`
- 唯一键建议：`db_id + alert_type + metric_key + direction`
- 同类活跃告警不重复推送，状态变化才推送

#### 4.2.3 降噪策略

- 冷却窗口（默认 15 分钟）
- 维护静默窗口（按实例/类型）
- 批量聚合推送（同窗口多实例同类告警）

### 4.3 基线与异常检测模块

#### 4.3.1 基线模型

- 时间槽：7 x 24 = 168 槽
- 窗口：近 28 天（可配）
- 统计量：mean/std/p95/p99/normal_range

#### 4.3.2 三重判定

1. 量级：超出正常区间
2. 方向：按指标方向（上升/下降/双向）
3. 持续性：连续 3 次满足才触发

#### 4.3.3 冷启动策略

- 数据不足时标记“基线未就绪”
- 回退固定阈值兜底，避免无告警能力

### 4.4 容量预测模块

#### 4.4.1 模型策略

- 数据不足：线性外推
- 有周期：Holt-Winters
- 波动明显：ARIMA（或等价时序模型）
- 按历史窗口回测选最优模型

#### 4.4.2 输出项

- 预计触达告警线日期（85%）
- 预计触达危险线日期（95%）
- 月增长率与置信度
- 扩容建议量与建议窗口

### 4.5 RCA 与优化建议模块

- 规则库驱动（事件 -> 原因 -> 建议）
- 支持复合规则组合推断
- 输出包含：
  - 可能根因（TopN）
  - 影响范围
  - 操作建议
  - 风险等级与回滚建议

### 4.6 工单与审批执行模块

- 风险分级：low/medium/high/critical
- 审批流：按风险等级动态路由
- 执行要求：
  - 前置检查
  - 幂等执行
  - 后置验证
  - 失败回滚
- 审计记录：审批人、执行人、时间、命令、结果、证据

### 4.7 可视化与报表模块

- 总览看板：可用性、活跃告警、高风险实例
- 单库详情：指标趋势 + 基线带 + 事件时间线
- 容量视图：到期时间排序与扩容建议
- 报表体系：日报/周报/月报/年度规划报告

## 5. 详细设计：数据设计

### 5.1 数据分层

1. 元数据层（配置、用户、权限、工单）
2. 时序层（原始指标点、聚合指标）
3. 事件层（告警、通知、审计、执行记录）
4. 智能层（基线模型、预测结果、评分结果）

### 5.2 核心表建议（逻辑）

- `database_config`
- `collection_snapshot`
- `metric_point`
- `alert`
- `alert_notification_log`
- `baseline_model`
- `prediction_result`
- `health_score`
- `audit_log`
- `remediation_execution_log`

### 5.3 数据保留策略

- 原始指标：12-18 个月
- 小时聚合：36 个月
- 日聚合：长期
- 告警/审计：>= 5 年（建议 7 年）

## 6. 详细设计：接口设计

### 6.1 内部接口

- 采集器接口：`check(config) -> metrics`
- 告警接口：`fire_or_resolve(event)`
- 基线接口：`get_current_baseline(metric, slot)`
- 预测接口：`predict(metric_key, horizon_days)`

### 6.2 外部 API（示例）

- `GET /api/v1/databases`
- `GET /api/v1/databases/{id}/status`
- `GET /api/v1/databases/{id}/metrics`
- `GET /api/v1/databases/{id}/baseline`
- `GET /api/v1/databases/{id}/prediction`
- `GET /api/v1/alerts`
- `POST /api/v1/alerts/{id}/acknowledge`
- `POST /api/v1/auditlogs/{id}/approve`
- `POST /api/v1/auditlogs/{id}/execute`
- `GET /api/v1/health`

接口要求：

- Token 认证 + 角色鉴权 + 数据域控制
- 标准错误码和审计 ID 透传

## 7. 详细设计：安全与合规

### 7.1 安全基线

- 禁止生产环境 `DEBUG=True`
- 凭据全部环境变量化/密钥托管
- 敏感字段加密存储（AES-256-GCM）
- 传输链路强制 TLS

### 7.2 权限与审计

- RBAC（角色）+ 数据域隔离（业务域/环境）
- 高风险操作二次确认或双人审批
- 审计日志不可篡改，支持追溯和导出

### 7.3 合规要求

- 关键操作全留痕
- 审批链路闭环
- 访问最小权限
- 定期安全巡检与权限回收

## 8. 详细设计：部署与运维

### 8.1 部署架构建议

- Web 层：Django + Gunicorn + Nginx（可双实例）
- 任务层：Celery Worker + Beat
- 存储层：PostgreSQL + TimescaleDB、Redis
- 通知层：邮件 + 钉钉（双通道）

### 8.2 平台自监控

必须监控平台自身：

- 任务积压、采集耗时、超时率
- 告警发送成功率、重试率
- API 延迟、错误率
- 数据库写入延迟与失败率

### 8.3 灾备指标

- 控制平面 RTO <= 15 分钟
- 告警状态 RPO <= 5 分钟
- 至少季度级故障演练（通道故障、队列积压、实例宕机）

## 9. 实施路线图

### 阶段 A（0-4 周）：稳态与安全

- 修复采集链路稳定性问题
- 完成配置安全改造与运行产物治理
- 建立最小自动化测试集

### 阶段 B（5-10 周）：架构升级

- 引入队列化调度与多 worker
- 完成 PostgreSQL/TimescaleDB 迁移
- 完成告警治理与多通道重试

### 阶段 C（11-16 周）：智能与闭环

- 上线基线增强、容量预测、RCA 升级
- 上线审批执行闭环与报表体系

### 阶段 D（17-24 周）：平台化增强

- 多租户策略治理
- 模型治理（灰度/漂移/回滚）
- 外部系统集成（CMDB/ITSM/IAM）

## 10. 验收标准

### 10.1 功能验收

- 支持 200+ 实例稳定接入
- 告警、审批、审计链路完整
- 报表可按周期自动产出

### 10.2 性能验收

- 5 分钟窗口内全量采集达标
- P1 告警延迟达标
- API P95 延迟达标

### 10.3 质量验收

- 关键模块测试覆盖达标（采集/告警/审批/加密）
- 无高危安全缺陷
- 关键故障演练通过

---

## 11. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 冷启动误报 | 告警信任度下降 | 基线未就绪回退阈值 + 明确标识 |
| 多库兼容性差异 | 采集失败上升 | 按版本矩阵做 SQL 兼容测试 |
| 通道不稳定 | 告警丢失 | 双通道 + 重试 + 补偿重放 |
| 并发压力增长 | 采集超时 | 分片扩容 worker + 压测调优 |
| 安全配置不当 | 合规风险 | 基线检查 + 发布门禁 |

---

## 12. 结论

本方案在你提出的三大建设方向上形成了完整闭环：

- 从“看见问题”升级为“提前预判问题”
- 从“人工经验驱动”升级为“规则 + 数据 + 模型协同驱动”
- 从“单点工具”升级为“可持续运营的平台能力”

建议以“稳态优先、智能渐进、闭环落地”的方式执行，先完成核心链路可靠性和安全合规，再逐步兑现高阶智能能力。
